# Страница благодарности: блок оплаты в Telegram «не закрывается»

## Уточнённая проблема

Речь не о редиректе с `/checkout`, а о **[странице благодарности](src/app/(shop)/checkout/success/page.tsx)**: пользователь видит ссылки/QR на оплату в Telegram, и **блок с этим призывом не исчезает**, когда уже не нужен (чаще всего — **после успешной оплаты**).

## Как сейчас устроено

Файл: [`src/app/(shop)/checkout/success/page.tsx`](src/app/(shop)/checkout/success/page.tsx).

- При валидном `token` показывается карточка «Оплата» с бейджем статуса.
- Если есть `paypassTelegramLink`, **всегда** выводятся текст, кнопка «Открыть оплату в Telegram» и QR — **в том числе при `paymentStatus === "paid"`**, потому что условие только `paymentLink ? (кнопка + QR) : (текст про недоступность)`.

Итог: после оплаты заказ может быть помечен как оплаченный (бейдж «Оплачен»), но **крупный блок с повторным переходом в бота остаётся** — воспринимается как «окно не закрылось».

## Целевое поведение

1. **До оплаты** (`pending` / частично `unpaid` при догрузке ссылки): показывать понятный блок — кнопка в Telegram + при необходимости QR, короткий текст что делать дальше.

2. **После оплаты** (`paid`): **не** показывать кнопку и QR; оставить компактное сообщение вроде «Оплата получена, спасибо» и при необходимости номер заказа / ссылку в чат поддержки.

3. **При отказе** (`failed`): бейдж + понятный следующий шаг (написать в чат / связаться с менеджером), без дублирования того же CTA что до оплаты, если это вводит в заблуждение.

4. **Опционально (улучшение)**: пока статус `pending`, периодически синхронизировать статус с PayPass ([`syncOrderPaymentStatusById`](src/lib/paypass-sync.ts)) на клиенте или через `router.refresh()` после таймера, чтобы пользователь **не обновлял страницу вручную** — тогда блок с Telegram сам сменится на «оплачено» без перезагрузки.

## План правок (минимальный обязательный объём)

- В [`success/page.tsx`](src/app/(shop)/checkout/success/page.tsx) внутри карточки «Оплата»:
  - Рендерить ссылку + QR **только если** статус ещё требует оплаты (например `pending` или `unpaid` при наличии ссылки), **а не** при `paid` (и при желании скорректировать для `failed`).
  - При `paid` — один короткий абзац под бейджем, без повторного призыва открыть оплату.

## Опционально

- Небольшой клиентский компонент «Обновить статус оплаты» или тихий polling раз в N секунд только пока `pending`, с вызовом server action / route handler, который дергает существующую синхронизацию и затем `refresh()` — чтобы закрыть сценарий «оплатил в Telegram, вернулся на вкладку — всё ещё висит оплата».

## Что не решается с сайта

- Закрыть **приложение Telegram** или вкладку после перехода по `tg://` / `t.me` нельзя из веб-страницы; фокус — на том, что **сама страница благодарности** не должна бесконечно показывать уже ненужный блок оплаты.
